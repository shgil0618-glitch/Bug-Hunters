#### 🍟 트러블슈팅
```
<사례1>  
- 문제: 정지 기간이 만료된 사용자가 자동으로 활성 상태로 복구되지 않는 문제 발생  
- 원인: 정지 만료 조건이 쿼리에서 명확히 처리되지 않아 만료된 데이터가 정상적으로 업데이트되지 않음  
- 해결: MyBatis Update 쿼리에 날짜 비교 조건을 명확히 적용, DTO–MyBatis 매핑 불일치 추적  
- 성과: 데이터 정합성 확보, 계층 간 매핑 오류 추적 능력 강화  
- 학습: 단순 UPDATE 오류를 넘어 계층 간 데이터 흐름 관리 역량을 습득  
- 향후 개선 계획: 사용자 상태 관리 로직을 스케줄러 기반 자동화로 확장 검토 

<사례2>  
- 문제: 상태 변경 실패 시에도 메일 발송 로직이 실행되지 않는 구조적 문제 발생  
- 원인: 사용자 상태 변경과 메일 발송 로직이 결합되어 있어 오류 발생 시 부가 기능이 실행되지 않음  
- 해결: 사용자 상태 변경 성공 여부를 기준으로 메일 발송 로직 실행, 비즈니스 로직과 부가 기능 분리  
- 성과: 오류 추적 체계화, 서비스 안정성 및 운영 효율성 강화  
- 학습: 비즈니스 로직과 부가 기능 분리 설계의 중요성을 체득  
- 향후 개선 계획: 이벤트 기반 아키텍처(Event Listener) 도입을 통해 상태 변경과 알림 기능 독립 운영
```
